# THE GARDEN STALL
## Document 12 — Player Knowledge, Notebook and Skill Tree Specification

**Working Title:** The Garden Stall  
**Document Type:** Player Learning / Notebook / Skill Tree / Customer Memory Specification  
**Status:** Draft 001  
**Purpose:** Define how the player learns plants, customers, locations, suppliers, pairings, mistakes, and retail judgement over time; how the notebook/codex records discoveries; and how the skill tree and special unlocks improve information, interpretation, and selected player abilities.

---

# 1. Purpose of This Document

This document defines the player knowledge system for The Garden Stall.

The Garden Stall is not only a game about buying and selling plants. It is also a game about gradually learning how plant retail works.

The player should learn:

- which plants sell to which customers
- which plants fit which locations
- which stock needs careful handling
- which plants should be reduced early
- which displays convert browsers into buyers
- which supplier offers are trustworthy
- which pairings work
- which customer descriptions point toward which plant needs
- which mistakes are useful lessons
- which staff and upgrades will help next

This learning should not live only in the player’s memory.

The game should record important discoveries in a notebook/codex system.

The core player knowledge question is:

> “What have I learned from selling plants, and how can I use that knowledge tomorrow?”

---

# 2. Core Principle

Knowledge is progression.

The player should become better because they genuinely understand more about the business.

Progression should not only mean:

- more money
- bigger van
- better shelves
- more suppliers
- more staff

It should also mean:

- better judgement
- better hints
- better customer reading
- better stock selection
- better display choices
- better supplier scepticism
- better advice
- better use of mistakes

The player should feel:

> “I’m not just richer. I’m actually getting better at this.”

---

# 3. Notebook Tone

The notebook should begin as a rough trader’s notebook and gradually become richer.

## 3.1 Early Notebook Feel

At the start, the notebook should feel:

- scrappy
- practical
- handwritten in spirit
- functional
- based on experience
- slightly rough like the starting stall

Example early note:

> Petunias: traditional outdoor shoppers seem to like these. Keep trays full.

## 3.2 Later Notebook Feel

As the stall develops, the notebook may become:

- more organised
- more illustrated
- more complete
- more confident
- closer to a plant trader’s bible
- part scrapbook, part codex, part business manual

The tone should remain grounded and useful.

It should not become overly whimsical or decorative before the player has earned that charm.

---

# 4. Notebook Unlock Principle

Buying a plant unlocks basic information.

Selling, advising, displaying, reducing, or failing with that plant unlocks deeper information.

Recommended model:

> Buying unlocks the page. Experience fills it in.

This keeps the notebook connected to play.

## 4.1 Basic Unlock

When the player first buys stock, the notebook creates a basic entry.

Example:

> Tomato Plants — basic page created.

Basic page may show:

- name
- category
- basic tags
- current owned stock
- general use
- very simple care warning

## 4.2 Experience Unlocks

The page gains deeper notes when the player:

- sells the plant
- sells it at a specific location
- sells it to a specific customer type
- sells it as part of a pairing
- reduces it successfully
- fails to sell it
- lets it degrade
- overprices it
- completes a special request using it
- gives good or bad advice about it

This makes the notebook a record of play, not a static encyclopaedia.

# 8. Individual Customer Identities

Customers are not only abstract archetypes.

Each game should generate a finite pool of individual customer identities.

These identities may appear repeatedly.

Each customer has:

- unique identity
- archetype weighting
- preferences
- memory of past interactions
- possible quirks
- possible secondary information
- satisfaction history
- relationship/reputation link

Not every customer needs a special quirk.

But some should.

This supports the feeling of a local trading world.

## 8.1 Customer Identity Fields

Possible hidden fields:

- customer ID
- name
- broad archetype
- preferred plant categories
- budget range
- common locations
- garden/flat status
- experience level
- gift-buying frequency
- pet ownership flag, if relevant
- sunlight/shade situation
- taste preference
- quirk, optional
- previous purchases
- previous satisfaction
- trust level

## 8.2 Customer Quirks

Examples:

- always asks vague questions
- buys for elderly parent
- has a shady balcony
- has cats
- loves purple flowers
- dislikes strong scent
- always looks for bargains
- buys herbs for cooking
- wants child-friendly plants
- kills houseplants often
- buys gifts in a hurry

Quirks should affect play only when relevant.

They should not make every customer into a full RPG character.

## 8.3 Customer Memory

Returning customers may remember:

- good advice
- bad advice
- unsuitable sale
- fair pricing
- good bargain
- stock they bought before
- whether the plant survived or pleased them
- whether they trust the stall

Example:

> Mrs. Hanley remembered the strawberry pot you suggested and came back for compost.

Customer memory should support local warmth and reputation without becoming too complex.

# 12. Mistakes and Lessons

Failures should be recorded as learning.

The notebook should not only celebrate success.

It should help the player convert mistakes into better judgement.

## 12.1 Mistake Types

Possible lesson entries:

- overpriced common stock
- poor location fit
- tired stock displayed too prominently
- lily sold without safety warning
- thirsty plant left dry
- reduced too late
- compost loaded for wrong location
- customer request answered too quickly
- wrong plant for shade/sun
- gappy trays left during rush
- poor supplier special offer bought

## 12.2 Mistake Entry Example

> Lesson learned: Tired stock needs honest pricing.  
> Past-peak lilies damaged the gift display today. Reduce them earlier or keep them out of the front area.

## 12.3 Mistakes Should Feel Useful

A mistake should not only punish.

It should produce:

- report feedback
- notebook note
- possible future hint
- customer memory
- unlock progress, where suitable

This makes the player less afraid of experimentation.

---

# 13. Real-World Knowledge Level

Notebook knowledge should be grounded but simplified.

The game should not become formal horticultural training.

Use practical, playable advice.

## 13.1 Good Notebook Advice

Examples:

> Mint is best recommended in a pot because it spreads strongly in the ground.

> Basil dislikes drying out and can look miserable quickly.

> Lilies are poor gifts for homes with cats.

> Petunias and lobelia are familiar basket partners.

> Tomatoes need sun and regular watering.

This is enough to feel real without overwhelming the player.

## 13.2 Avoid

Avoid early over-detail such as:

- exact soil pH
- detailed fertiliser schedules
- Latin taxonomy overload
- complex pruning rules
- disease diagnosis
- advanced propagation
- botanical lectures

The notebook should support the selling game.

---

# 14. Mystery Versus Clarity

The game should not reveal everything immediately.

By default, the player should receive rough hints rather than exact hidden formulas.

## 14.1 Default Mode

Reveal:

- broad customer patterns
- useful notes
- likely plant fits
- broad location hints
- practical lessons

Do not reveal:

- exact customer scores
- exact hidden demand percentages
- exact sale formula
- full tag weights

## 14.2 Difficulty Settings Later

Later difficulty modes may change clarity.

### Relaxed

- more direct hints
- more visible tags
- stronger recommendations
- easier notebook unlocks

### Standard

- broad hints
- normal unlocks
- readable but not exact

### Expert

- fewer hints
- less explicit matching
- more reliance on player judgement
- slower notebook filling

Prototype 0.1 should use Standard-style hints.

# 17. XP and Unlock Model

Use a hybrid model:

> Achievements unlock nodes. General XP buys them.

This prevents the player from grinding one easy action to unlock unrelated knowledge too early.

## 17.1 General XP

The player earns XP from:

- completing trading days
- profitable days
- special request success
- discovery unlocks
- strong display performance
- good stock care
- good reduced stock recovery
- spotting bad supplier offers
- returning customer satisfaction

General XP can be spent on unlocked nodes.

## 17.2 Achievement Gates

Specific achievements unlock relevant skill nodes.

Examples:

- sell 20 petunias
- sell 10 tomato plants
- complete 5 gift requests
- clear 10 reduced plants
- trade 3 strong days at Village Green
- spot 3 bad special offers
- sell 5 plant+pot pairings
- keep display Good or better for a full day
- sell a vague customer the perfect plant

These gates ensure knowledge follows experience.

# 18. Special Unlock Skills

Not all skills are on the visible skill tree.

Some are special unlocks earned through specific achievements or memorable moments.

These skills may unlock directly rather than requiring normal tree purchase.

They should feel like discoveries or breakthroughs.

## 18.1 First Prototype Special Unlock

The first and easiest special unlock should be:

## Ask a Follow-Up

Unlock condition:

> Sold a vague customer the perfect plant.

Effect:

> During certain special requests, the player may ask one follow-up question to reveal secondary information.

This should be available in the test prototype.

## 18.2 Why This Skill Matters

Many real customer requests begin vague.

Example:

> “I want something nice for the front.”

With Ask a Follow-Up, the player can ask:

- “Is it sunny or shady?”
- “Is it for a pot or the ground?”
- “Is it a gift?”
- “Do they have pets?”
- “Do you want flowers now or something longer-lasting?”

The answer reveals secondary information and improves recommendation accuracy.

This makes special requests more interactive without turning them into long dialogue trees.

---

# 19. Ask a Follow-Up Specification

## 19.1 Usage

When a suitable customer request appears, the UI shows:

> Ask a follow-up

The player may select one question from a small list.

The game reveals secondary information.

## 19.2 Example

Customer says:

> “I need something colourful for outside.”

Available follow-ups:

1. “Is it sunny or shady?”
2. “Is it for a pot or a border?”
3. “Do you want it to last or just look good now?”

Customer answer:

> “It’s a shady doorstep, really.”

This changes the best recommendation.

## 19.3 Limits

Prototype rule:

- one follow-up per eligible request
- not every request needs it
- follow-up consumes time or attention
- staff may later handle simple follow-ups
- better Customer Reading skills may unlock stronger follow-up choices

## 19.4 Unlock Moment

When unlocked:

> New skill unlocked: Ask a Follow-Up  
> Some customers do not know how to ask for what they need. You can now ask one extra question during vague requests.

This should feel like the player learned how to sell, not like a magical ability.

Special requests should be one of the strongest learning sources.

The notebook should record:

- customer wording
- inferred need
- chosen recommendation
- outcome
- better option, if failed and known
- follow-up question used, if any

# 32. Knowledge and Balance Risks

## Risk 1: Skill Tree Becomes Magic Buffs

If skills mostly boost numbers, the player may stop learning.

Mitigation:

- most skills reveal information or improve interpretation
- bonuses must be small and thematic

## Risk 2: Notebook Becomes Homework

If entries are too long or technical, players may ignore them.

Mitigation:

- practical notes
- short summaries
- progressive reveal
- useful examples

## Risk 3: Too Much Hidden Simulation

If the game hides too much, sales feel random.

Mitigation:

- end-of-day feedback
- notebook discoveries
- clear customer comments
- skill unlocks that reveal patterns

## Risk 4: Too Much Exact Information

If everything is revealed, the game becomes formulaic.

Mitigation:

- broad hints by default
- exact detail reserved for dev overlay or difficulty settings
- customer individuality adds softness

## Risk 5: Failure Feels Bad

If mistakes only punish, experimentation stops.

Mitigation:

- mistakes create notebook lessons
- failed requests can still unlock clues
- customer trust can reward honest correction

# 37. Success Criteria

The knowledge system succeeds if:

- the player feels they are learning the business
- notebook entries are useful
- mistakes become lessons
- customer quirks make the world feel local
- skill unlocks feel earned through experience
- Ask a Follow-Up improves request play
- plant/customer/location connections become clearer over time
- information progression feels more important than raw buffs
- the player checks the notebook willingly
- end-of-week learning feels as satisfying as profit

The system fails if:

- notebook entries are ignored
- skill unlocks feel arbitrary
- too much information is shown too early
- too little information makes sales feel random
- customer identities become busywork
- skills become generic stat buffs
- failures do not teach anything
- the system feels detached from stock, display, and selling

# 39. Final Knowledge Definition

The Garden Stall’s knowledge system turns experience into progression.

The player begins with a rough notebook and limited understanding. Buying stock creates basic pages. Selling, failing, reducing, advising, pairing, and returning customers fill those pages with practical knowledge. Customers are both broad archetypes and finite individual identities with possible quirks. Locations reveal their patterns through repeated trading. Mistakes become lessons. Pairings become named combos. Skills mostly improve information and interpretation, with some small thematic bonuses. Special unlocks, beginning with Ask a Follow-Up, reward memorable player achievements.

The core question is:

> “What did today teach me that will make tomorrow’s stall better?”
